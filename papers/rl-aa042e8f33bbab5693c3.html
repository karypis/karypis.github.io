Teaching Language Models to Think in Code #25 · score 72empirical

arxiv.org · 📄 Open PDF
**TL;DR** - **What it does:** This paper introduces THINC (Thinking in Code), a training framework that restructures tool-integrated reasoning so that code itself serves as the primary reasoner — replacing the interleaved natural-language/code paradigm — through trajectory distillation, supervised fine-tuning, and reinforcement learning, producing THINC-4B which achieves 78.1% avg@16 across five competition-level math benchmarks. - **Why it's important:** Interleaved tool-integrated reasoning systems routinely let NL arithmetic errors propagate silently into code, use code only as a post-hoc verifier rather than a solver, and duplicate work between NL and code; these structural flaws corrupt final answers and degrade robustly under execution failures, limiting the ceiling of LLM mathematical reasoning. - **How it does it:** THINC constrains every trajectory to a single NL planning block followed by code-only reasoning blocks connected solely through interpreter execution outputs, then trains this format via 12.2k distilled trajectories (SFT) and three-stage GRPO/DAPO reinforcement learning with verifiable rewards. - **The good:** THINC-4B achieves 78.1% avg@16 across five competition benchmarks, outperforming all TIR baselines including the much larger rStar2-Agent-14B and ReTool-32B, surpassing Qwen3-235B-A22B-Thinking on four of five benchmarks, and maintaining 64–69% recovery@3 under consecutive code failures versus near-collapse for interleaved baselines. - **The missing:** All experiments are confined to 1.7B and 4B parameter models and competition-level mathematics; whether the code-centric format transfers to larger scales or other tool-integrated domains (e.g., web search, multi-tool agentic tasks) is entirely untested. - **Read it or not:** Read it if you work on mathematical reasoning, tool-integrated LLM training, or RL post-training — the structural diagnosis of interleaved TIR is sharp and the ablation against ASTER-4B (matched base, teacher, and RL pipeline) isolates the trajectory format as the source of gain.

TL;DR - What it does: This paper introduces THINC (Thinking in Code), a training framework that restructures tool-integrated reasoning so that code itself serves as the primary reasoner — replacing the interleaved natural-language/code paradigm — through trajectory distillation, supervised fine-tuning, and reinforcement learning, producing THINC-4B which achieves 78.1% avg@16 across five competition-level math benchmarks. - Why it's important: Interleaved tool-integrated reasoning systems routinely let NL arithmetic errors propagate silently into code, use code only as a post-hoc verifier rather than a solver, and duplicate work between NL and code; these structural flaws corrupt final answers and degrade robustly under execution failures, limiting the ceiling of LLM mathematical reasoning. - How it does it: THINC constrains every trajectory to a single NL planning block followed by code-only reasoning blocks connected solely through interpreter execution outputs, then trains this format via 12.2k distilled trajectories (SFT) and three-stage GRPO/DAPO reinforcement learning with verifiable rewards. - The good: THINC-4B achieves 78.1% avg@16 across five competition benchmarks, outperforming all TIR baselines including the much larger rStar2-Agent-14B and ReTool-32B, surpassing Qwen3-235B-A22B-Thinking on four of five benchmarks, and maintaining 64–69% recovery@3 under consecutive code failures versus near-collapse for interleaved baselines. - The missing: All experiments are confined to 1.7B and 4B parameter models and competition-level mathematics; whether the code-centric format transfers to larger scales or other tool-integrated domains (e.g., web search, multi-tool agentic tasks) is entirely untested. - Read it or not: Read it if you work on mathematical reasoning, tool-integrated LLM training, or RL post-training — the structural diagnosis of interleaved TIR is sharp and the ablation against ASTER-4B (matched base, teacher, and RL pipeline) isolates the trajectory format as the source of gain. Problem & Motivation: Mathematical reasoning in LLMs has increasingly relied on tool-integrated reasoning (TIR), where NL thought blocks alternate with code execution across multiple turns. Practitioners in LLM post-training and competition-math solving face three recurring failure modes: (1) the model completes a derivation in NL, then runs code only to confirm — code adds no new computation; (2) when NL performs arithmetic, errors propagate as hard-coded constants into subsequent code blocks without detection; (3) NL and code duplicate each other's reasoning roles, with NL laying out the algorithm step-by-step and code merely transcribing it. These structural problems are intrinsic to the interleaved TIR format rather than being model-specific failures, and prior RL or prompting interventions have not addressed them at the trajectory-format level. Key Idea: The key insight is that code is already a symbolic reasoning system equivalent to mathematics in expressive power, so NL reasoning between code blocks is structurally redundant and error-prone. THINC eliminates NL from the reasoning chain entirely after an initial strategic planning step: the first block (t1) expresses only a high-level solution strategy, and every subsequent block is code whose inputs are exclusively the execution outputs of prior code blocks. This means every intermediate value is produced and verified by the interpreter by construction, NL errors cannot propagate, and the roles of planning (NL) and computation (code) are cleanly separated. Methodology: The pipeline has three stages. First, trajectory distillation: using a 3-shot prompt demonstrating the THINC format, the authors sample one trajectory per problem from Qwen3.5-27B on Skywork-OR1 and OpenMathReasoning (English, positive-integer-answer problems). Trajectories are retained only if correct, error-free in every code block, contain ≥3 code blocks, and spend <50% of tokens in the planning thought block (|t1|/|τ| < 0.5). This yields 12,200 THINC-SFT trajectories. Second, supervised fine-tuning: Qwen3-1.7B and Qwen3-4B-Thinking-2507 are fine-tuned on THINC-SFT using the standard next-token prediction objective (m_k = 1 for all tokens), context length 32K, learning rate 7×10⁻⁶ with cosine schedule, batch size 16, 3 epochs. Third, reinforcement learning: starting from SFT checkpoints, GRPO is applied on DAPO-Math-17k using the DAPO variant with token-level normalization and asymmetric clipping (ε_low = 0.20, ε_high = 0.28), no KL penalty, rollout batch size 128, G = 8 trajectories per prompt. Training runs three curriculum stages of increasing context budget and tool call limit (Stage 1: 280 steps, 16K context, 20 tool calls; Stage 2: 120 additional steps filtering trivial problems; Stage 3: from step 400, 32K context, 40 tool calls). The reward function is binary exact-match. Key design decisions: (1) retaining all tokens in the SFT loss (including execution outputs) — authors find no significant difference from masking them; (2) strict filtering criteria that enforce the code-centric format at distillation time rather than relying on RL alone; (3) the per-code-block stationless design where variables do not persist between turns, forcing re-imports and hardcoded values from prior results. Key Results: THINC-4B achieves 78.1% ± 4.4% avg@16 across AIME 2024 (88.3%), AIME 2025 (85.8%), AIME 2026 (86.0%), HMMT 2025 February (74.0%), and BeyondAIME (56.1%). This is the best result in the comparison table, outperforming all TIR baselines: ASTER-4B (73.8%), rStar2-Agent-14B (61.5%), ReTool-32B (50.7%), DemyAgent-4B (57.3%), CoRT-1.5B (57.0%). It also surpasses the NL-only Qwen3-235B-A22B-Thinking (75.2%) on four of five benchmarks, and outperforms its own distillation teacher Qwen3.5-27B (64.7% with 3-shot) by 13.4 points despite being 7× smaller. The most controlled ablation compares THINC-4B against ASTER-4B, which shares the same base model, teacher capacity, and RL pipeline but uses an interleaved trajectory structure: THINC-4B exceeds ASTER-4B by 4.1 points on average with fewer tool calls (6.1 vs. 11.1) and shorter responses (13.5k vs. 15.4k tokens). Code-grounding analysis shows 99.2% of THINC-4B final answers appear in at least one code block's execution output, versus 88.4% for ReTool and 74.3% for rStar2. Recovery@k (fraction reaching correct answer after k consecutive initial code failures) shows THINC-4B maintains 64–69% recovery for k = 1,2,3, dropping to 54.5% at k = 4 and 33.3% at k = 5, while ASTER drops from 52.1% to 18.5% and rStar2-Agent collapses to 0% at k = 5. OOD evaluation on GPQA-Diamond shows THINC-4B at 66.48% avg@16 and 91.41% best@16, compared to ASTER-4B at 63.42/90.40 and base Qwen3-4B-Thinking at 66.32/83.84. THINC-1.7B reaches 42.8% avg@16, outperforming Qwen3-1.7B (32.2%), the tool-prompted base (29.8%), and CoRT-1.5B (25.7%). Novelty Classification: NEW METHOD + NEW DATASET/BENCHMARK. The genuinely novel contribution is the trajectory format design: restricting NL to a single strategic planning step and routing all subsequent reasoning through code-only blocks connected by execution outputs. The individual training components — trajectory distillation from a teacher model, SFT, GRPO/DAPO RL with verifiable rewards — are established techniques applied from prior TIR and post-training literature. The THINC-SFT dataset of 12,200 trajectories is a new artifact. The combination of format design + the empirical demonstration that this structural change explains the performance gap (via the ASTER-4B ablation) constitutes a substantive contribution beyond applying known methods to a new setting. Rigor & Reproducibility: The paper's key ablation — comparing THINC-4B against ASTER-4B under matched conditions (same base model, same teacher, same RL pipeline, differing only in trajectory format) — is the strongest piece of evidence isolating the format as the causal variable; this is methodologically sound. Error bars (95% confidence intervals) are reported throughout. The code-grounded answer rate (99.2%) and Recovery@k metric are well-designed behavioral analyses that go beyond accuracy numbers. No significant ablation is provided to show which of the three SFT filtering criteria (correctness, error-free execution, ≥3 code blocks, planning token ratio) individually matter — the filtering criteria are justified by intuition but not ablated. The claim that m_k = 1 vs. m_k = 0 (masking execution tokens) produces no significant difference is stated but not shown in detail. Code and models are declared as "will be released soon" — not yet publicly available at submission time. Training used 8× NVIDIA H200 GPUs. Multiple random seeds for RL training are not mentioned explicitly. This is a preprint; results have not undergone peer review. Deployability (4-property check): Profile: EMPIRICAL (training framework + trained models evaluated on benchmarks). - Practical: Training on 8× NVIDIA H200 GPUs is commercially accessible in cloud settings; inference runs within a 32K token budget using standard Python interpreter access (numpy, scipy, sympy). - Efficient: THINC-4B uses fewer tool calls (6.1 vs. 11.1 for ASTER) and generates shorter responses (13.5k vs. 15.4k tokens), giving lower inference cost than comparable interleaved baselines at the same model size. - Effective: 78.1% avg@16 on five competition-level benchmarks with 99.2% code-grounded answers provides strong evidence the method solves the stated problem of unreliable NL-mediated computation. - Scalable: Experiments are confined to 1.7B and 4B parameter models; whether the format and training recipe hold at 7B, 14B, or 70B+ is not tested and constitutes the single most important open question. The paper delivers two trained model checkpoints (pending release) and a 12.2k SFT dataset; no public library or CLI tool is provided. Strengths: - The three-failure-mode diagnosis of interleaved TIR (post-hoc verification, NL error propagation, role confusion) is precise and falsifiable, not merely motivational. - The ASTER-4B ablation is unusually clean: matching base model, teacher, and RL pipeline isolates trajectory format as the variable, providing stronger causal evidence than most TIR papers. - Recovery@k is a novel and informative metric that reveals robustness properties invisible to accuracy-only evaluation; the result that THINC maintains 64–69% recovery while interleaved systems degrade to near-zero is a concrete behavioral characterization. - The OOD evaluation on GPQA-Diamond shows the code-centric format generalizes beyond mathematical training distribution, which partially addresses the scope concern. Author-Acknowledged Limitations: The authors explicitly state two limitations: (1) experiments are restricted to 1.7B and 4B parameter models due to computational constraints; (2) evaluation is confined to competition-level mathematics. They name extension to larger models and other tool-integrated domains as natural future directions. They confront these limitations directly rather than minimizing them, which is credibility-positive. Analyst-Identified Gaps: - The SFT dataset is drawn from Skywork-OR1 and OpenMathReasoning, which overlap significantly with competition math benchmarks — the degree of train/test contamination across AIME and HMMT problems is not analyzed. - THINC's stateless code execution (each block is a fresh Python process, variables do not persist) is an unusual design choice that forces models to hardcode prior results; this may harm performance on problems requiring long stateful derivations, but no analysis of failure cases where statefulness would have helped is provided. - The SFT filtering criteria (particularly the 50% planning-token ratio threshold) are engineering choices presented without ablation; the sensitivity of downstream RL performance to these thresholds is unknown. - The comparison includes Qwen3-235B-A22B-Thinking, a much larger NL-only model, but does not compare against any large TIR system (e.g., a 70B TIR model); the ceiling of the interleaved TIR paradigm at large scale is therefore not established. - Code and models are not yet released, making independent replication impossible at time of publication. Bottom Line: Any practitioner building or training tool-integrated mathematical reasoning systems should read this paper — the ASTER-4B ablation credibly demonstrates that the trajectory format alone (not model size or RL recipe) drives a meaningful accuracy gain, and the Recovery@k analysis reveals a robustness advantage that standard evaluation metrics miss entirely.
